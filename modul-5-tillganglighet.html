Lektion 1: Tillgänglighet – alt-texter, kontrast och tabbordning

CSS-spaning — NYTT: :focus-visible och skip-link
Tryck Tab nu — ser du en blå ram hoppa runt länkarna? Det är :focus-visible i aktion. Skillnaden mot vanlig :focus: med :focus visas ramen även när du klickar med musen (irriterande). :focus-visible är smartare — den visar ramen bara när webbläsaren bedömer att du navigerar med tangentbordet. Prova också att trycka Tab direkt när sidan laddats — en röd "Hoppa till huvudinnehållet"-länk dyker upp högst upp. Det är en skip-link: den är gömd med top: -100px och hoppar till top: 0 vid :focus.

Varför håller vi på med det här? Tillgänglighet låter som någonting som bara berör en liten grupp — men räkna: ca 2% av befolkningen är blinda eller gravt synskadade, ca 8% av alla killar har någon form av färgblindhet (bland tjejer är det under 1%), och massor av folk navigerar med tangentbord (av biomekaniska skäl, preference, eller för att de är pro-användaré). Det är inte en liten grupp. Det är din mormor, din kompis med tendinit, och en miljard andra.

Alt-text — inte "bild", utan vad bilden faktiskt visar

Alt-texten på en bild är det en skärmläsare läser upp. Om du skriver dålig alt-text är bilden lika hjälplös som om den inte fanns alls.

Bildtyp Dålig alt-text Bra alt-text
Profilbild alt="bild" alt="Johan Eriksson, lärare på Teknikgymnasiet"
Produktbild alt="skor" alt="Nike Air Max 90 i svart och vitt, storlek 42"
Graf/diagram alt="diagram" alt="Stapeldiagram som visar att mobilanvändning ökade med 40% 2023"
Dekorativ bild alt="vacker bakgrundsbild" alt="" (tom! skärmläsaren hoppar över den)
Logotyp alt="logo" alt="Spotify" (bara företagets namn)
Tumregel: Testa om din alt-text funkar genom att läsa den högt utan att titta på bilden. Förstår du vad bilden föreställer? Bra. Om det låter konstigt — skriv om. 
<!-- Dekorativ bild — tomt alt-attribut, skärmläsaren hoppar över den -->
<img src="bakgrund.jpg" alt="">

<!-- Informativ bild — beskriv vad den visar -->
<img
  src="diagram-mobilanvandning.png"
  alt="Stapeldiagram: mobilanvändning ökade från 60% till 84% mellan 2018 och 2023"
>

<!-- Länk med bild inuti — alt-texten beskriver länkdestinationen -->
<a href="startsidan.html">
  <img src="logotyp.svg" alt="Hem — tillbaka till startsidan">
</a>

Färgkontrast — WCAG AA kräver 4,5:1 för text

Kontrast mäts som ett förhållande — hur stor skillnad är det i ljusstyrka mellan text och bakgrund? WCAG AA (den nivå som är lag i Sverige för offentliga sajter) kräver:

Innehåll Minimikontrast (WCAG AA) Bättre kontrast (WCAG AAA)
Vanlig text (under 18pt) 4,5:1 7:1
Stor text (18pt+ eller 14pt fet) 3:1 4,5:1
UI-komponenter, ikoner 3:1 —

Visuellt exempel — samma text, helt olika läsbarhet:

DÅLIG — kontrast ca 1,6:1
Det här är text i grå färg på grå bakgrund. Svårt att läsa, eller hur?
BRA — kontrast ca 16:1
Det här är vit text på mörk bakgrund. Tydlig och lättläst.

Verktyg för att kolla kontrast:

  • WebAIM Contrast Checker — klistra in två färgkoder, får kontrastvärdet direkt
  • Chrome DevTools — klicka på en färg i CSS-panelen, det dyker upp ett kontrastvärde automatiskt
  • Figma och liknande designverktyg har inbyggd kontrastkoll

Tabbordning — hur tangentbordsnavigation funkar

När en användare trycker Tab hoppar fokus mellan interaktiva element (länkar, knappar, formulärfält) i den ordning de förekommer i HTML-koden. Om din HTML-struktur är logisk funkar tabbordningen automatiskt rätt. Problem uppstår när man försöker fixa det med CSS eller tabindex på fel sätt.

<!-- tabindex="0" — gör ett icke-interaktivt element fokuserbart (använd sparsamt) -->
<div tabindex="0" role="button">Klicka på mig</div>

<!-- tabindex="-1" — elementet är fokuserbart via JavaScript men inte via Tab -->
<div id="modal" tabindex="-1">...</div>

<!-- tabindex="1" eller högre — UNDVIK. Bryter den naturliga ordningen. -->
<a href="#" tabindex="5">Konstigt hopp i tabbordningen</a>

<!-- BÄSt: låt HTML-ordningen styra. Inga tabindex behövs. -->
<nav>
  <a href="/">Hem</a>       <!-- Tab 1 -->
  <a href="/om">Om oss</a>  <!-- Tab 2 -->
</nav>
<main>
  <a href="/artikel">Läs mer</a>  <!-- Tab 3 -->
</main>

:focus-visible vs :focus — vad är skillnaden?

/* Gammalt sätt — :focus */
a:focus {
  outline: 3px solid blue;
  /* Visas ALLTID när elementet har fokus — även vid musklick.
     Många designers tyckte det såg fult ut och tog bort det helt.
     Det är ett tillgänglighetsproblem! */
}

/* Modernt sätt — :focus-visible */
:focus-visible {
  outline: 3px solid #0db5e6;
  outline-offset: 3px;
  /* Visas BARA när webbläsaren bedömer att du navigerar med
     tangentbord/switch/annat icke-mus-verktyg.
     Vid musklick — ingen ring. Vid Tab — tydlig ring. */
}

/* Ta bort default-ramen, ersätt med din egen */
:focus:not(:focus-visible) {
  outline: none;
}

Skip-link — för att slippa tabba igenom hela menyn

Föreställ dig att du måste trycka Tab 30 gånger för att ta dig förbi navigationsmenyn innan du kan läsa innehållet — på varje sida. En skip-link löser det: den hoppar rakt till <main>.

<!-- Placera detta som FÖRSTA element i <body> -->
<a class="skip-link" href="#huvudinnehall">Hoppa till huvudinnehållet</a>

<!-- ...din navigation... -->

<main id="huvudinnehall">
  <!-- Här börjar innehållet -->
</main>
/* CSS för skip-link */
.skip-link {
  position: absolute;
  top: -100px;        /* Gömd utanför skärmen */
  left: 16px;
  background-color: #e94560;
  color: #fff;
  padding: 10px 18px;
  border-radius: 0 0 6px 6px;
  font-weight: 700;
  z-index: 9999;
  transition: top 0.2s ease;
}

.skip-link:focus {
  top: 0;             /* Glider ner när den får fokus */
}
Vanligt misstag: Att dölja skip-linken med display: none eller visibility: hidden fungerar INTE — då kan skärmläsare inte hitta den. Använd position: absolute; top: -100px istället — den är visuellt gömd men fortfarande i DOM-trädet.

Uppdrag: Kolla din senaste sida

  1. Öppna ditt senaste projekt i Chrome.
  2. Stäng av musen och navigera med bara Tab och Enter. Funkar det? Ser du var du är hela tiden?
  3. Högerklicka på en textbit → "Inspektera" → klicka på färgen i CSS-panelen → kolla kontrastvärdet som dyker upp.
  4. Kolla dina bilder — har alla informativa bilder en beskrivande alt-text (inte bara "bild" eller "img")?
  5. Lägg till en skip-link överst på sidan.

Lektion 2: W3C och WCAG – standarder och riktlinjer

CSS-spaning — NYTT: @media (prefers-reduced-motion) och @media (prefers-color-scheme)
Dessa är media queries för användarpreferenser — de reagerar på inställningar i operativsystemet, inte skärmstorlek. Spinnen nedan är animerad. Om du har aktiverat "Minska rörelse" i ditt OS (Windows: Inställningar → Hjälpmedel → Visuella effekter; Mac: Systeminställningar → Hjälpmedel → Bildskärm) stoppas animationen automatiskt via @media (prefers-reduced-motion: reduce). Rutans utseende längre ner i lektionen ändras om ditt OS är i ljust läge — via @media (prefers-color-scheme: light). Inget JavaScript behövs.

Vad är W3C?

W3C (World Wide Web Consortium) är organisationen som bestämmer hur webben ska fungera — tekniskt sett. De skriver specifikationerna för HTML, CSS och en massa annat. När webbläsare (Chrome, Firefox, Safari) stöder samma standard kan du skriva kod en gång och ha den fungera överallt.

Tim Berners-Lee — mannen som uppfann webben 1989 — grundade W3C 1994 och var dess director fram till 2022. Ganska bra CV.

Varför spelar standarder roll? Utan dem skulle varje webbläsare göra sin grej. Det är delvis vad som hände på 90-talet — Netscape och Internet Explorer hade olika HTML, och webbutvecklare fick göra dubbla versioner av allt. Standards = du skriver en gång, det funkar överallt.

WCAG 2.2 — fyra principer

WCAG (Web Content Accessibility Guidelines) är W3C:s riktlinjer för tillgänglighet. Version 2.2 publicerades i oktober 2023 och är den aktuella versionen. Allt bygger på fyra principer — kom ihåg dem med förkortningen POUR:

Princip Engelska Vad det betyder
Möjlig att uppfatta Perceivable All information måste kunna uppfattas — inte bara med synen. Text, bilder med alt-text, undertexter på video.
Hanterbar Operable Allt måste gå att använda utan mus. Tangentbord, switch, pekare — alla ska funka.
Begriplig Understandable Innehåll och gränssnitt måste vara förståeligt. Tydliga felmeddelanden, förutsägbart beteende.
Robust Robust Innehållet måste funka i olika webbläsare och hjälpmedel, nu och i framtiden. Validera din HTML.

Nivåer: A, AA och AAA

WCAG delar in krav i tre nivåer:

  • A — Lägsta nivån. Måste uppfyllas. Utan den är sidan nästan oanvändbar för många.
  • AA — Mellannivån. Det här är vad lagen kräver för offentliga webbplatser i Sverige (DOS-lagen). Rimlig att uppnå.
  • AAA — Högsta nivån. Bäst möjlig tillgänglighet. Inte alltid realistiskt för hela sidor, men sträva mot det.

Vanliga WCAG-krav och hur du möter dem

Krav Nivå Hur du gör det
Alt-text på bilder A alt="beskrivning" på alla informativa bilder, alt="" på dekorativa.
Tangentbordsnavigation A Allt som går att klicka ska gå att nå med Tab och aktivera med Enter.
Sidtitel A Varje sida måste ha en beskrivande <title>.
Synlig fokusindikator AA Använd :focus-visible med tydlig outline. Ta inte bort outline helt.
Textkontrast 4,5:1 AA Kolla med WebAIM Contrast Checker eller Chrome DevTools.
Rubriker i logisk ordning A h1 → h2 → h3. Hoppa inte från h1 till h4.
Undertexter på video A Lägg till <track kind="captions"> i din <video>-tagg.
Ingen autoplay-animation som blinkar A Blinkande innehåll kan trigga epileptiska anfall. Undvik det.
Textkontrast 7:1 AAA Sträva mot detta om möjligt — alla vinner på högre kontrast.

Validering — W3C Validator

Ogiltig HTML kan bete sig konstigt i skärmläsare och andra hjälpmedel. W3C erbjuder gratis validatorer:

  • validator.w3.org — HTML-validator. Klistra in din kod eller ange en URL.
  • jigsaw.w3.org/css-validator — CSS-validator.
Varför validera? Webbläsare är förlåtande — de försöker visa sidan även om HTML:en är trasig. Skärmläsare är det inte alltid. En saknad stängningstagg eller ett felaktigt nästlat element kan bryta navigationen för en skärmläsaranvändare.

@media prefers-reduced-motion — respektera rörelsepreferenser

Vissa användare mår illa av animationer — det gäller folk med vestibulära störningar (balansproblem), men också folk som helt enkelt gillar ett lugnt gränssnitt. Operativsystemen har inbyggda inställningar för "minska rörelse" — CSS kan läsa av dem.

Den här ringen snurrar — om du inte har aktiverat "Minska rörelse" i ditt OS. 

/* Animation finns som default */
.spinner {
  animation: snurra 1s linear infinite;
}

/* Om användaren vill ha lugnt gränssnitt: stäng av */
@media (prefers-reduced-motion: reduce) {
  .spinner {
    animation: none;
  }

  /* Generell regel — stäng av alla animationer */
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

@media prefers-color-scheme — ljust och mörkt läge

De flesta moderna OS har ett mörkt och ett ljust läge. CSS kan anpassa sig automatiskt — utan JavaScript, utan inloggning, utan cookies. Det är också en tillgänglighetsgrej: många med dyslexicertifiering eller migrän föredrar specifika kontrastsätt.

Den här rutan byter utseende beroende på om ditt OS är i ljust eller mörkt läge. Prova att byta i dina systeminställningar! 
/* Bas: mörkt läge (om man bygger dark-first) */
body {
  background-color: #0f0f1a;
  color: #dde1f0;
}

/* Ljust läge — aktiveras automatiskt om OS är i light mode */
@media (prefers-color-scheme: light) {
  body {
    background-color: #ffffff;
    color: #1a1a2e;
  }

  .kort {
    background-color: #f0f4ff;
    border-color: #ccc;
  }
}

/* Alternativt — CSS custom properties (förhandstitt på Modul 7!)
   Med variabler (--bg, --text) behöver man bara ändra på ett ställe */
:root {
  --bg: #0f0f1a;
  --text: #dde1f0;
}

@media (prefers-color-scheme: light) {
  :root {
    --bg: #ffffff;
    --text: #1a1a2e;
  }
}

body {
  background-color: var(--bg);
  color: var(--text);
}
Förhandstitt: Koden med --bg, --text och var() använder CSS custom properties (CSS-variabler) — ett koncept som tas upp ordentligt i Modul 7. Det viktigaste nu är att förstå grundmönstret med @media (prefers-color-scheme).

Uppdrag: WCAG-jakt

  1. Gå till validator.w3.org och validera din senaste HTML-fil. Fixa eventuella fel.
  2. Leta upp tre tillgänglighetsbrister på en valfri webbsida med Chrome DevTools → Lighthouse → Accessibility. Skriv upp vad du hittade.
  3. Lägg till @media (prefers-reduced-motion: reduce) i ditt projekt för alla transition och animation du använder.
  4. Boss-level: Implementera @media (prefers-color-scheme) med CSS custom properties i ditt projekt. Byt theme baserat på OS-inställning.

Lektion 3: Praktisk tillgänglighetskontroll i tidigare projekt

CSS-spaning — NYTT: @media print och prefers-contrast
Tryck Ctrl+P (eller Cmd+P) för att öppna utskriftsdialogen — välj "Förhandsgranska" utan att faktiskt skriva ut. Se hur sidan ser helt annorlunda ut: mörk bakgrund borta, vit bakgrund, skip-link och navigationsknappar dolda, URL:er skrivs ut efter länkarna. Det är @media print i aktion. Det andra nya: @media (prefers-contrast: more) — om du aktiverat "Öka kontrast" i ditt OS (Mac: Systeminställningar → Hjälpmedel → Bildskärm) ser sidan ännu skarpare ut. Inte för att det är nödvändigt här — men som ett mönster att känna till.

Lighthouse — det kraftfullaste gratis-verktyget du redan har

Lighthouse är inbyggt i Chrome DevTools och analyserar din sida på fyra axlar: Performance, Accessibility, Best Practices och SEO. Du får en poäng 0–100 och en lista med konkreta åtgärder.

  1. Öppna din sida i Chrome och tryck F12 (eller högerklicka → Inspektera).
  2. Klicka på fliken Lighthouse (kan vara gömd bakom >>-pilen).
  3. Kryssa i Accessibility (och gärna de andra) — klicka Analyze page load.
  4. Vänta ca 10–30 sekunder medan Lighthouse kör sin analys.
  5. Läs igenom resultaten. Varje problem har en länk till dokumentation om hur du fixar det.
  6. Fixa de enklaste felen, kör om analysen — se poängen gå upp.
Mål: En tillgänglighetspoäng på 90+ är bra. 100 är möjligt men kräver ofta extra arbete. Sikta på minst 80 i dina projekt — och förstå varför poängen är vad den är.

Manuell testning — Lighthouse missar en del

Lighthouse hittar ca 30–40% av tillgänglighetsproblem automatiskt. Resten kräver manuell testning. De här testerna tar 5 minuter och hittar saker som ingen algoritm kan:

Test Hur du gör det Vad du letar efter
Tangentbordstest Koppla från musen, navigera med Tab, Shift+Tab, Enter, piltangenter. Kan du nå allt? Ser du alltid var fokus är? Fastnar du i någon loop?
Zoom 200% Ctrl++ (eller Cmd++) tills du är på 200%. Bryts layouten? Försvinner text bakom kanter? Rullningslist horisontellt?
Skärmläsare (Windows) Aktivera NVDA (gratis) eller Windows Narrator (Win+Ctrl+Enter). Läses innehållet upp i logisk ordning? Är bildtexter vettiga?
Skärmläsare (Mac) Aktivera VoiceOver med Cmd+F5. Samma som ovan.
Färgblindhetstest Chrome DevTools → Rendering-fliken → Emulate vision deficiencies. Förstår man fortfarande innehållet utan att förlita sig på färg?

Checklista — det du ska kunna bocka av

Gå igenom varje punkt för ditt projekt:

  • HTML validerar utan fel på validator.w3.org
  • <html lang="sv"> är satt (eller rätt språk)
  • Varje sida har en unik, beskrivande <title>
  • Rubriker är i logisk ordning: h1 → h2 → h3 (inga hopp)
  • Semantiska taggar används: <header>, <nav>, <main>, <footer>
  • Alla informativa bilder har beskrivande alt-text
  • Dekorativa bilder har alt=""
  • Textkontrast är minst 4,5:1 för vanlig text
  • Synlig fokusindikator finns för alla interaktiva element
  • Hela sidan går att navigera med bara tangentbord
  • Skip-link finns och fungerar
  • Formulärfält (om du har) har kopplade <label>-taggar
  • Lighthouse Accessibility-poäng: 80 eller högre
  • @media (prefers-reduced-motion: reduce) stänger av animationer

Vanliga problem och hur du fixar dem

Problem Lighthouse hittar Felet i koden Fix
"Image elements do not have [alt] attributes" <img src="foto.jpg"> Lägg till alt="beskrivning" eller alt="".
"Links do not have a discernible name" <a href="..."><img src="pil.svg"></a> Lägg till aria-label="Nästa sida" på länken, eller alt-text på bilden.
"Form elements do not have associated labels" <input type="text" placeholder="Namn"> <label for="namn">Namn</label><input id="namn" type="text">
"Heading elements are not in a sequentially-descending order" <h1> följt av <h4> Använd h1 → h2 → h3 i ordning. Hoppa inte nivåer.
"Background and foreground colors do not have a sufficient contrast ratio" Ljusgrå text på vit bakgrund. Mörkare text eller ljusare bakgrund. Kolla med WebAIM Contrast Checker.
"Buttons do not have an accessible name" <button><img src="sok.svg"></button> <button aria-label="Sök"><img src="sok.svg" alt=""></button>

@media print — skriv ut som ett proffs

Nästan ingen tänker på det, men folk skriver ut webbsidor. En CV-sida, en recept-sida, en informationssida — alla kan hamna på ett papper. Utan print-CSS blir det kaos: mörk bakgrund, navigationsmenyer, knappar som inte gör någonting på papper.

@media print {
  /* Dölj det som inte behövs på papper */
  nav,
  .skip-link,
  .css-spaning,
  button {
    display: none;
  }

  /* Ljus bakgrund, mörk text (standard för papper) */
  body {
    background-color: #ffffff;
    color: #000000;
    font-size: 12pt;
  }

  /* Visa URL:er i klartext efter länkarna */
  a[href^="http"]::after {
    content: " (" attr(href) ")";
    font-size: 10pt;
    color: #555;
  }

  /* Undvik sidbrytningar mitt i rubriker */
  h2, h3 {
    page-break-after: avoid;
  }

  /* Börja varje stor sektion på ny sida */
  section, article {
    page-break-before: always;
  }
}

@media prefers-contrast — för de som behöver det riktigt skarpt

/* Om OS är inställt på "Öka kontrast" */
@media (prefers-contrast: more) {
  body {
    background-color: #000000;
    color: #ffffff;
  }

  a {
    color: #ffff00;         /* Gul — syns tydligt på svart */
    text-decoration: underline;
  }

  button {
    border: 2px solid #ffffff;
  }

  /* Inga halvtransparenta bakgrunder */
  .kort {
    background-color: #000000;
    border: 2px solid #ffffff;
  }
}

/* Om OS är inställt på "Minska kontrast" (ovanligare) */
@media (prefers-contrast: less) {
  body {
    color: #555;      /* Mjukare kontrast för de som föredrar det */
  }
}

Projekt: Revidera ett tidigare projekt

Välj ett av dina projekten från modul 1–4. Gör en fullständig tillgänglighetsrevision:

  1. Kör Lighthouse — skriv upp poängen och alla fel.
  2. Gå igenom checklistan ovanför punkt för punkt.
  3. Fixa minst 5 problem du hittar — börja med de enklaste (alt-texter, rubriker).
  4. Lägg till skip-link, :focus-visible-styling och @media (prefers-reduced-motion).
  5. Lägg till ett print-stylesheet med @media print.
  6. Kör Lighthouse igen — hur mycket gick poängen upp?
  7. Redovisa: Vad var det värsta problemet du hittade? Vad var lättast att fixa? Vad var svårast?

Boss-level: Skärmläsartest

  1. Aktivera NVDA (Windows, gratis på nvaccess.org) eller VoiceOver (Mac, Cmd+F5).
  2. Navigera din sida med bara tangentbordet medan skärmläsaren läser upp.
  3. Är det meningsfullt? Hoppar den rätt? Läser den upp bilderna korrekt?
  4. Skriv ner tre saker du vill förbättra.
CSS-sammanfattning — Modul 5: Det här är vad vi introducerade i den här modulen:
  1. Lektion 1: :focus-visible (tangentbordsfokus utan musklick), :focus:not(:focus-visible) (ta bort standardring), skip-link med position: absolute; top: -100px som glider till top: 0 vid fokus.
  2. Lektion 2: @media (prefers-reduced-motion: reduce) (stänger av animationer för känsliga användare), @media (prefers-color-scheme: light/dark) (anpassar färgtemat till OS-inställning).
  3. Lektion 3: @media print (utskriftsanpassning — dölj nav, ljus bakgrund, skriv ut URL:er), @media (prefers-contrast: more) (förstärkt kontrast för de som behöver).

Allt annat på sidan — mörkt tema, kodrutor, tabeller, info-rutor, flexbox, grid, @import av typsnitt, transition — det är från modul 1–4 och återanvänds här utan att introduceras på nytt.
